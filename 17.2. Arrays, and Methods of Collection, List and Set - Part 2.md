#### The Collection Interface
Java's "Collections API" provides several standard data structures declared in some core interfaces, along with some implementations of those interfaces

###### `Iterable` 
represents "many" of something
undefined size
no particular rules about order, duplicates, nulls
no (direct) ability to update
`forEach`, `iterator`, `spliterator` use in `enhanced for loop`.

- ###### `Iterable.Iterator`
`hasNext`, `next`, `[remove]`

###### `Collection` (sub interface of `Iterable`)
defined size
ability to add, remove, and query elements

###### `List` (sub interface of `Collection`)
user-specified order is maintained
enhanced, bidirectional, iteration (ListIterator)
supports sorting and sub-lists
indexed access for add, get, and remove
find position of contained item

###### `Set` (sub interface of `Collection`)
rejects duplicates (add method returns boolean)
order managed internally:
- Independent of user operations
- supports fast searching

###### `Map` interface
key-value pairs (aka "dictionary", "associate array")

###### `Map.Entry`
pair, or tuple, of one key with its associated value

#### Requirements for Set Elements
Sets should provide fast lookup, this requires internal organization and support from the elements to be added.
Two common organization techniques are supported by Java's core APIs: hashing and ordering.
Items to be added to a `HashSet` (or `HashMap`) will be lost if they do not implement meaningful `equals` and `hashCode` methods.
To add items to a `TreeSet` a valid ordering must be provided.
Mutating an item that's been added to any set is likely to result in corruption of the set structure.

##### Quick Question
Given:
```java
class Student {
	String name;
	public Student(String name) {
		this.name = name;
	}
}
```
and
```java
Set<Student> ss = new HashSet<>();
Student f1 = new Student("Fred");
ss.add(f1);
ss.add(f1);
ss.add(new Student("Fred"));
System.out.println(ss.size());
``
```
What happens?

Answer: prints 2 
Adding `f1` a second time fails (`equals`  delegate to `==`)
Adding the `new Student("Fred")` succeeds, since this is *not* dot-`equals` to `f1`.

`-------------------------`
What happens?
```java
List<Integer> li = new ArrayList<>(List.of(1, 2, 3, 4));
li.remove(1);
System.out.println(li);
```

Answer: `[1,3,4]`
`remove(int)` removes by `index`
`remove(Integer.valueOf(1))` would result in `[2,3,4]`

`-------------------------`
What happens?
```java
HashSet<String> hs = new HashSet<>(Set.of("Fred", "Jim"));
hs.add("Sheila");
TreeSet<CharSequence> tcs = new TreeSet<>(Set.of("Fred"));
tcs.add("Sheila"); tcs.add("Jim");
System.out.println(hs.equals(tcs));
```
Answer: `true`
Both sets contain the same elements (order is irrelevant)
The `equals` and `hashCode` methods for Sets should return consistent values even across different type s of set.
