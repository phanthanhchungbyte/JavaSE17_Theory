#### Core Lambda Facts
Java's lambda expressions are *object* expressions that
implement an *interface*
- They are instances of a nameless, autogenerated, class
	- but there is no `this` context created by that instance
- The desired interface must be *unambiguously* known from the *context* of the lambda expression
- The interface must have *exactly one abstract* method
	- Perhaps annotated `@FunctionalInterface`
- A lambda can *only* implement that one abstract method
	- so, there are still good uses for anonymous inner classes

#### Basic Lambda Syntax
Before lambda:
```java
class X123 implements Predicate<String> {
	@Override
	public boolean test(String s) {
		return s.length() > 3;
	}
	Predicate<String> longString = new X123();
}
```

After lambda:
```java
Predicate<String> longString = (String s) -> {
	return s.length() > 3
};
System.out.println(longString.test("Hello"));
```

#### Lambda Syntax Variations
If the formal parameter type is unambiguous, it can be omitted
```java
(s) -> {return s.length() > 3;}
```

If the formal parameter type is unambiguous,  `var` can be used
```java
(var s) -> {return s.length() > 3;}
```

If there is *exactly one* formal parameter, and no type is given (neither explicit nor `var`), the parentheses can be omitted
```java
(s) -> {return s.length() > 3;}
```

If the method body has only a single statement, and that's a return, the body can be replaced by the expression returned.
```java
s -> s.length() > 3
```

#### The Type of Formal Parameters
The formal parameters for a lambda expression
- may all have explicit types
```java
(LocalDate ld, String s) -> body
```
- may all use the var pseudo-type
```java
(var ld, var s) -> ... body
```
- or may all be inferred
```java
(ld, s) -> body....
```
- however, combinations are prohibited
```java
(ld, var s) -> ... body ...
(LocalDate ld, s) -> ... body...
(LocalDate ld, var s, t) -> ... body...
```

Using `var` or explicit types  allows annotations on the parameters
```java
(@NotNull var s, var t) -> ... body ...
```

#### Matching the Signature
A lambda expression is a shorthand for an object that implements a specific method in a known interface
- The argument count, and any explicit types, must match the signature of that method
- The return type must be compatible, too
	- autoboxing/unboxing and widening promotions are permitted
	- expression statements are void compatible

#### Expression Statements
Expression statements are:
- assignment
- increment/decrement operations
- method invocation
- object instantiation

This means that an expression lambda that must have a void return type may actually have any of those expression types.
Examples:
```java
Consumer<String> cs = s -> aSet.add(s);
Consumer<String> cs2 = s -> new String();
```

##### Quick Question
Assume `Student` has a `getGrade` method, and that `PStud` is an `@FunctionalInterface` with an abstract method that takes a single `Student` argument and returns boolean.

Which of these are well-formed?
```java
PStud ps = Student s -> { return s. getGrade() > 3; };
PStud ps = s -> { s.getGrade() > 3 };
PStud ps = s -> { return s.getGrade() > 3; };
PStud ps = (s) -> s.getGrade() > 3;
PStud ps = s -> s.getName().length();
```

Answer: Only 3 and 4 is correct.


Assume `BFSS` is a functional interface with an abstract method with `Student` and `String` arguments and returns `String`.
Which of these are well-formed?
```java
BFSS f = (Student s, str) -> { return str + ": " + s.getGrade();};
BFSS f = (@NotNull s, str) -> str + ": " + s.getGrade();
BFSS f = s, str -> str + ": " + s.getGrade();
BFSS f = (Student s, String str) -> str + ": " + s.getGrade();
BFSS f = (@NotNull var s, var str) -> str + ": " + s.getGrade();
```

Answer: Only 4 and 5 is correct
