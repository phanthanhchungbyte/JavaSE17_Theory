#### General Execution Requirements
When starting a program, three things will be needed
- the entry point where the program starts
- where to find the modules
- which modules are needed

The entry point is specified using the `--module (or -m)` directive
- specify module `-name/fully.qualified.ClassName` that contains the `main` method
	`--module test.module/my.pkg.MyClass`

#### Finding Modules
Modules will be found by looking in the list of directories
enumerated in the `--module-path (-p)` specification
- this is the same as for the compilation phase
- use the platform specific separator

Looking ahead, a module specifies the other modules upon
which is is dependent
- this is done using a `requires` directive

#### Which Modules are Needed?
When the JVM starts, it identifies all needed modules
- the result is referred to as the module graph
- structurally the dependencies form a directed acyclic graph
- identification starts with the module containing the main class/method
	- then adds all the modules this module requires
	- continue transitively to build a set of all the modules mentioned in requires directives

#### Patching the Module Graph
When a program includes code not written for the module system, it might be necessary to patch the module graph
- Non-modular code has no `module-info.java` and therefore cannot declare requires
- To work around this, necessary modules might have to be added manually using the `--add-modules` directive

#### Adding Non-Modular Code
Non modular code may be supplied by placing a regular jar file on the module path
- this creates an *automatic module*
- this *must be a jar file*, it cannot be in a simple directory structure
- if a manifest entry `Automatic-Module-Name` exists then this defines the name of the module
- alternatively, the name is taken from the jar's base filename minus what looks like version information
	`my.library.1.2.3-SNAPSHOT.jar -> my.library`

#### Important Notes About Class Loading
In the module system, a class can be loaded from a module *only* if that module is in the module graph
- it's not sufficient for the class to be in a module that's on the module path
	- this mechanism speeds the search for classes
	- and hence reduces the startup time of modular applications relative to non-modular ones
- The classpath can still exist
	- it's accessible from an automatic module
	- but is *inaccessible* from regular modules

#### More About Automatic Modules
If a module needs code from an automatic module, it must use a `requires` directive, just as if the target were a regular module
- Regular modules cannot access classes on classpath
	- however, automatic modules can
	- as such, they form a "bridge/
- An automatic module cannot use the `requires` directive
	- they read everything (including classpath)
	- (hence the need for `--add -modules`)
	- however, they can only access module contents with the consent of that module

##### Quick Question
The `main` method is in `UseIt` in `b.mod`, that method needs code from `legacy.jar` which does not have a `module-info` file. Also, `legacy.jar` needs code from `a.mod`, but `b.mod` does not. How will you run this code?

Tree
```tree
=extras/
==legacy.jar
=modules/
==a.mod/
===module-info.class
===pkg/
====mult/
=====Multiplier.class
==b.mod/
===module-info.class
===pkg/
====UseIt.class
```

Answer:
1) `legacy.jar` can be treated as an automatic module
2) To define the entry point, we need
	`--module b.mod/pkg.UseIt`
3) To find the modules, we'll need a module path that includes *both* `modules` and `extras`
	`--module-path modules:extras`
4) The `module-info` in `b.mod requires legacy`, so the automatic module will be in the graph
5) `requires a.mod` is missing from this, so we need `--add-modules a.mod`
