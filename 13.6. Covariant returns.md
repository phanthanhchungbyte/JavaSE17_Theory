#### Liskov Substitution
When one method overrides another, it should not break the expectations created by the original method

This is the *Liskov Substitution Principle*
Java's compiler verifies syntax aspects of this requirement
- accessibility
- checked exceptions declared
- and return type

#### Acceptable Return Types
When overriding a method that returns a reference type
- return any assignment compatible type
When overriding a method that returns a primitive type
- return the *same* type

##### Quick Question
What would be valid and invalid return types when overriding this method?
```java
class P {
	Number doStuff() {return 1;}
}
```
Answer: 
- Valid examples, `Number, Integer, Long, BigDecimal, AtomicInteger`
- Invalid examples, `short, int, long, String`


What would be valid and invalid return types when overriding this method?
```java
class P {
	long doStuff() {return 1;}
}
```
Answer: *only* `long` (primitive) is valid
