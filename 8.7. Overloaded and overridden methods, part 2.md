#### Method Overriding
Generally, if a method has the exact same name, and the same argument type sequence as a method in a parent type
- and both methods are instance methods then this is an attempt at *overriding*
or
- if both methods are static and the parent is a class, then this is an attempt at *hiding*

If both methods are in the same class, it's a failed attempt at overloading!

#### Restrictions on Overriding/Hiding
If a method is marked `final`, it cannot be overridden nor hidden
- The compiler rejects the attempt

#### Rules for Overriding / Hiding Methods
In overriding, the subclass method is seen as a "substitute" for the parent method
-  Because `aParent` might refer to a `Child` instance, invoking an overridden behavior might invoke the `Child's` code
- The *compiler* must decide how the call will behave based on the `Parent` type
- But... the actual behavior might be that of the `Child`
The *Liskov substitution principle* suggests avoding "surprises" in these situations

#### The Liskov Substitution Principle in Java
To conform with the Liskov substitution principle, the Java compiler will ensure that for an overloading method,
- The return type must "conform"
- no *unexpected* checked exceptions should arise
- the method must be at least equally accessible
- generic types must conform
Note that the same rules apply to the case of hiding a method in a parent class (that is, static methods in the parent class and child class)
- However, these rules do not apply if the parent method is a static defined in an `interface`

#### Conforming Return Types
The return type of an overriding method must be either
- assignment compatible if a reference type
or
- exactly the same if a primitive type

#### Late/Dynamic Binding
If a reference variable of `Parent` type refers to an object of `Child` type, when an overridden instance method is invoked
- the behavior is that defined in `Child` class
This is sometimes called late binding, or dynamic binding, and is a key feature of object-oriented polymorphism
- In Java this behavior *only* happens with *overloaded instance method invocation*; not static methods, and not with any kind of field.

##### Quick Question
<span style="color: #6495ED;">If this method signature is declared in a parent class</span>
```Java
long doStuff()
```
<span style="color: #6495ED;">Which would be valid in a child class in the same package?</span>
```Java
A) long doStuff() throws RunTimeException // not checked exception, so ok
B) int doStuff()
C) Integer doStuff()
D) String doStuff(int x) throws SQLException
E) long doStuff()
```
Answer: A, D, E. D is a overload
`=============================`
<span style="color: #6495ED;">If this  method signature is declared in a parent class</span>
```Java
CharSequence doStuff()
```

<span style="color: #6495ED;">Which would be valid in a child class in the same package?</span>
```Java
A) Object doStuff()
B) CharSequence doStuff() throws SQLException
C) String doStuff()
D) char[] doStuff()
```
Answer: C

`==========================`
<span style="color: #6495ED;">If this method signature is declared in a parent class</span>
```Java
protected CharSequence doStuff()
```

Which would be valid in a child class in the same package?
```Java
A) public CharSequence doStuff()
B) private CharSequence doStuff()
C) protected String doStuff()
D) String doStuff()
E) private CharSequence doStuff(int x)
```
// Must not be less accessible in the subclass while overriding
Answer: A, C, E

#### One More Thing
If you define a method in a class `Thing`:
```java
public boolean equals(Thing other)...
```
This is an *overload*, not an override!
- It will *not* be used by the collections for equivalence checks
But watch out for generic methods in generic classes, e.g.
```Java
class Thing implements Comparable<Thing> {
	@Override
	public int compareTo(Thing o) {return 0;}
}
```
