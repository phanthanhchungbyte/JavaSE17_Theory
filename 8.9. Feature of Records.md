#### `record` Methods
A record type can have custom and customized methods
- Automatic methods can be replaced
	- `equals, hashCode,toString,` and access methods
	- Replacement method must conform to expected signature
	- `@Override` annotation is permitted
	- No `super`-dot-xxx method will be available
- Additional methods can be added
	- Instance or static
	- No native or abstract methods are permitted 

#### `record` Fields
A record type cannot declare instance fields
- nor instance initializers
- Auto-generated fields match the components
However, it *can* have static fields and static initializers
- These need not be final; they play no part in equality, etc. of instances

#### `record` Initialization
The automatic internal instance fields of record types are final
- They must be properly and reliably initialized
This initialization is the task of the canonical constructor
- The canonical constructor can be automatic, or hand-coded
- It must no be less accessible than the class
	- Automatic version will have the same accessibility
- The argument list exactly matches the component list
	- Including the component name
- It must initialize the final fields that store the component values

#### Hand-Coding the Canonical Constructor
This explicit constructor is identical to the automatic one
```Java
record X(int a, String b) {
	X(int a, String b) {
		this.a = a;
		this.b = b;
	}
}
```

#### The Compact Constructor
The main reasons for explicit coding of the canonical constructor are validation and normalization
- A short form syntax is also provided to simplify this
- This form "rolls over" to the automatic canonical constructor on completion
- A compact constructor *must not* initialize the storage
```Java
record X (int a) {
	X {
		if (a < 0) throw new IllegalArgumentException();
	}
}
```

#### Additional Constructors
Regular classes often have multiple constructor overloads allowing initialization from differing argument lists allowing initialization from differing argument lists
- A common use case is provision of default values
This can be done with a record too:
```Java
record X(int a) {
	// automatic canonical constructor is created
	X() {this(99);}
	X(int p, int q) {this(p + q);}
}
```
- These constructors must reliably delegate to the canonical constructor

#### Additional Observations
Record types are prohibited from having instance fields and also from having instance initializers
- Instance initializers exist to initialize instance fields
Deserialization of a serializable record is performed using the canonical constructor
- This differs from normal serializable classes which do not use constructors

##### Quick question
Given
```Java
record Vessel(int loa, int displacement) {
	@Override
	public String toString() {
		return loa > 100 ? "Ship loa=" + this.loa : super.toString();
	}
}
```
and:
```Java
out.println(new Vessel(150, 75_000));
```
What happens?

`===========================`

What do you notice?
```Java
record Vessel(int loa, int displacement) {
	private final int tonnage;
	Vessel(int loa, int displacement) {
		this.tonnage = displacement / 2_000;
		this.loa = loa;
		this.displacement = displacement;
	}
	publid int tonnage() {
		return tonnage;
	}
}
```

Answer: The variable tonnage is prohibited - no instance fields may be declared in a record, except indirectly by the components

`==========================`
What do you notice, how might you improve it?
```Java
record Vessel (int loa, int displacement) {
	Vessel {
		this.loa = loa;
		this.displacement = displacement;
	}
}
```
Answer: 
- a compact constructor must not write to the fields created for the component values. A long form canonical constructor by contrast, must do so.
- In this case, the "best" improvement is simply to delete the constructor and let the auto-created canonical constructor do the work.
