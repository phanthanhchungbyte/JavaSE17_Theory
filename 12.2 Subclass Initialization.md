#### Initializing Subclass Instances
We have seen that, in any one class, instance initialization precedes execution of the constructor body

In a class hierarchy, Java seeks to ensure parent initialization precedes child initialization. To support this any constructor
- starts with a programmer-provided `this(...)` invocation
- ***or*** has a programmer-provided call to `super(...)`
- ***or*** a call to `super()` will be compiler generated

#### Additional Notes on Initialization
Constructor delegation cannot be recursive, or the compiler will complains

It's bad practice to invoke overridable instance methods in a constructor, since they might delegate to code in an uninitialized subclass-part of the current object.
- but the compiler permits this

##### Quick Question
<span style="color: #F0F8FF;">What happens?</span>
```Java
class Y {{System.out.println("initializing Y");}}
public class X extends Y {
	{System.out.println("Initializing X");}
	public X() {System.out.println("X()");}
	public X(int x) {
		this();
		System.out.println("X("+x+")");
	} 
	public static void main(String[] args) {
		new X(1);
	}
}
```
Answer:
`initializing Y`
`initializing X`
`X()`
`X(1)`

<span style="color: #F0F8FF;">What happens?</span>
```Java
class Y {{System.out.println("initializing Y");}}
public class X extends Y {
	{System.out.println("Initializing X");}
	public X() {
		this(1);
		System.out.println("X()");
	}
	public X(int x) {
		this();
		System.out.println("X("+x+")");
	} 
	public static void main(String[] args) {
		new X();
	}
}
```
Answer: Compilation fails due to recursive constructor calls
