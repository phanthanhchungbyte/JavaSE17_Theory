#### Declaring Bounded Type Parameters
A type parameter declaration can have multiple bounds
```java
class C1 {}
interface I1 {}
interface I2 {}
interface I3 {}
class B<E extends C1 & I1 & I2 & I3> {}
```
- Note the & `--` commas separate multiple *type* params
- Any class bound must be listed first
	- and only one is permitted
- The keyword `extends` is used to mean "is assignment compatible with"

#### Introducing Wildcards
Consider these methods
```java
public static void doStuffLC(List<CharSequence> lc) {} 
public static void doStuffLS(List<String> ls) {}
```
Which arguments should these methods accept?
```java
List<CharSequence> lc = ...;
List<String> ls = ...;
```
What will the methods do? Add items? Extract items?
- If we try to add `StringBuilder (a CharSequence)` to a `List<String>` that would corrupt the list 
- If we try to take items from a `List<CharSequence>` we might get a `StringBuilder`, but we expected only `String`

Consider again how this method might use its arguments
```java
public static void doStuffLC(List<CharSequence> lc) {}
```
*If* this method *only reads* items from the list and assigns them to a variable of type `CharSequence` then it *could* work with any list of things that *can be assigned* to `CharSequence`

We can assign an object to a `CharSequence` variable provided:
- it's of `CharSequence` type
- or any *subtype* of `CharSequence` (such as `String`)

Therefore, the method *could* work with any list of `CharSequence` or a sub-type of `CharSequence`

#### Upper Bounded Wildcard Syntax
To express a list that contains things that can be assigned to CharSequence:
```java
public static void doStuffLC(List<? extends CharSequence> lc) {}
```

Notice this method does not know the actual type of the items in the list (hence the question mark), only that they will be assignment compatible to CharSequence
- We can use them that way safely
- If we try to do anything else with them, the compiler will refuse

This is called an *upper bound*, and correlates to co-variance

#### The Simple Wildcard
Often we want to express the idea of `List` of anything
- This is *not* the same as `List<Object>`
- We cannot put anything in this list
- We only know that what comes out of the list will be some kind of object

Naturally, this requires the syntax:
```java
List<? extends Object>
```

However, this shorthand means the same thing:
```Java
List<?>
```

#### Introducing Wildcards # 2
Consider again how this method might use its arguments
```java
public static void doStuffLS(List<String> ls) {}
```
*If* this method only *adds* strings to the list then it *could* work with any list that "accepts" strings.

We can assign a `String` object to any variable of:
- `String` type
- or any *supertype* of `String` (i.e, `CharSequence`, or `Object`)

Therefore, the method could work with any list of string or a super-type of string

#### Lower Bounded Wildcard Syntax
To express a list that accepts `String`:
```java
public static void doStuffLC(List<? super String> ls) {}
```
Notice this method does not know the actual type of the items in the list (hence the question mark), only that `String` can be assigned to that type
- We can assign `String` objects to the list type safely
- If we try  to do anything else with the list type, the compiler will refuse

This is called a *lower-bound*, and correlates to contra-variance

#### Wildcards are not Declarations
Wildcards constrain the types that a *caller* may use
- they do not declare a type parameter
- this is usually just fine, since the code will be written in terms of the explicit type that is mentioned after `? extends` or `? super`

However, it's possible to declare a type parameter that's constrained by another type parameter, e.g.
```java
static <E, F extends E> E[] as Array(List<F> many, Class<E> base)
```

Note that `super` cannot be used in a type parameter declaration, this fails:
```java
<E, F super E>
```

##### Quick Question
Which are valid?
```java
static <E extends CharSequence> void show1(List<E> le) {}
static <? extends CharSequence> void show2(List<E> le) {}
static void show3(List<? extends CharSequence> le) {}
static void show1(List<E extends CharSequence> le) {}
static <E, F super E> void show1(List<E> le, list<F> lf) {}
```
Answer:
- 2 is wrong, a `?` cannot declare a type parameter
- `super` cannot be used to declare a type parameter

Two parts: Does this work? How should it be improved?

```java
interface Function<A, B> {

class C<A,B> {
	private A a;
	public void modify (Function<A, A> fn) {
		a = fn.apply(a);
```

Answer: Yes, it works

```java
public void modify(Function<? super A, ? extends A> fn)
```
