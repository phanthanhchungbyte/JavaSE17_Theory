#### Overview of Sealed Type Hierarchies
Sometimes a design benefits from defining a specific set of types that can ever be subtypes of a particular generalization
- If code must be provided for each of those types, unexpected new types could cause bugs
- "How we use this" vs. "How it does it"
	- Behavior external to the class(es)
	- Testing with `instanceof`
- To builds a controlled hierarchy of types
	- The root type must indicate permitted direct subtypes
	- Each subtype must indicate if it permits further subtypes, and if so, what

#### Core Rules of Sealed Type Hierarchies
As noted, some rules apply to sealed type hierarchies
- extends / implements clause(s) must precede permits
- A sealed type must identify* all the permitted sub-types
- The "leaf" nodes of the hierarchy will *commonly* be final**
	- A non-sealed node could be a leaf
- A non-leaf node can be sealed or non-sealed
	- If sealed, allowing explicit sub-types
	- If non-sealed allowing arbitrary sub-types
- An interface, abstract class, or class can all be sealed or non-sealed
- An interface or abstract class cannot be final

##### Quick Question
<span style="color: #F0F8FF;">What do you notice?</span>
```Java
sealed interface Vehicle permits Car, Truck extends Serializable {}
```
Answer: This does not compile, the permits clause must come after extends or implements clauses

<span style="color: #F0F8FF;">What happens here?</span>
```java
sealed abstract class Widget permits DoDah, ThingUmAJig {}
non-sealed class ThingUmAJig extends Widget {}
class DoDah extends Widget {}
```
Answer: The class `DoDah` fails to compile because it's not marked `sealed`, `non-sealed` nor `final`
The other non-sealed class doesn't change this

#### Locations of Sealed Hierarchy Members
If code is developed to run under the JPMS (Module System)
- Members of a sealed hierarchy may be in any package
	- but must be in the *same module*
But, if code runs without JPMS
- Members of a sealed hierarchy must all be in the *same package*
