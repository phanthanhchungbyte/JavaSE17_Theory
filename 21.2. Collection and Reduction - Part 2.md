#### Mutating Reduction
A pure function creates a new result object for every execution
- memory allocation and GC might become expensive

An alternative would be to mutate an intermediate result object
- other than the result, keep the operation pure
- this need not be unsafe or error prone if the object is kept in a tight scope and not shared
- if using associativity to group the operations, each group needs its own result object
- and a mutating aggregation must happen at the end

This type of operation is called *collection* in the stream API

#### The `collect` Method of Stream
The stream API provides a collect method that performs as outlined

For a stream type T and a result type R
```java
R collect(Supplier<R> supplier,
	BiConsumer R <? super T> accumulator,
	BiConsumer <R,R> combiner)
```

Notice in every case, the *output* side is the *first* argument to the accumulator and combiner operations
• This is critical in the combiner above, as the two arguments are the same type, so the compiler cannot check the order!
