#### Field Declaration Syntax
Fields in types have a general declaration form:
`<modifiers> Type Identifier [initialization]`
- Modifiers are annotations,  `public/protected/private, static, final, transient, and volatile`
	- No repetitions
	- Order is not significant (but is guided by convention)
- Type can include generic information
	- e.g.: `List<String> or List<E>`
- Initialization uses simple assignment from an expression
- For multiple variables of the same type, use a comma separated list of everything Right of Type

##### What happens?
```Java
public class Scratch<E> {
	List<E> le, lf = null, List<String> ls;
}
```
Answer: Compilation fails. Different types must have separate declarations


```Java
public class Scratch {
	final static public @Deprecated List<String> le;
}
```
Answer: Compilation fails, final variables must be definitely, explicitly, initialized

#### Array Field Declaration
Arrays have two declaration forms
```Java
int [] ia
int ia []
```
These can even be mixed
```Java
int [] ia2 [] = {{1,2}, {3}};
```

#### Default Initialization of Fields
Fields receive default initialization
- Value will be "zero-like" : `false, '\0',0,0.0,null`
- Elements of an array are fields, and initialized as such
- This is `not` sufficient for final fields
Default initialization occurs as an indivisible part of memory allocation
- Attempts to read fields before explicit initialization generally show the default value
- Curious rules apply if using *initializers*, to read a field before it has been *declared*

##### Quick Question
```Java
class X {
	static int x;
	public static void main(String[] args) {
		int y;
		System.out.println(x);	
		System.out.println(y);
	}
}
```
Answer: Compilation fails because y is not initialized

```Java
class X {
	static int x;
	public static void main(String[] args) {
		int y;
		System.out.println(x);	
	}
}
```
Answer: Prints 0

```Java
class Y {
	static final int x;
	static {
		System.out.println(x);
	}
}
```
Answer : Compilation fails because x is not initialized
