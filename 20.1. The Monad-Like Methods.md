Imagine we have some data in a list and a function that's applicable to the type of those items

We could apply that operation to every item in turn and put the results in a new list

#### Applying an Operation to Many Items
The "many" items need not be in a list, they could be in a one-at-a-time mechanism, such as an iterable.

In this case the result of the map operation might also be an iterable and items can be processed one at a time

![[Pasted image 20260517150538.png]]

In this situation, the processing elements (the map operation) are necessarily stateful, and can be used only once (like an iterator)

#### Applying an Operation to Many Items
The model described illustrates the execution of the `Stream` API
- It's often referred to as *lazy*
- Items are not pushed down the process pipeline, they are pulled through on request
- Many stages can be chained, creating a pipeline 
- In this simple mental model one item of data will proceed all the way along the pipeline before another starts the journey 
- The operation that does the pulling is usually the last operation in the pipeline, and gives a result instead of returning another `Stream` and allowing further chaining
- This type of processing permits an infinite stream

#### Methods in Stream Pipelines
Traditionally, the map method should not modify any data,
Instead the argument function should be *pure*
- It should not mutate any data visible outside the function 
	- note that function arguments are visible outside the call
- It should create a new item to represent changed state
- Its result should depend only on the argument values
This makes things easier to understand and test
- but it has *critical* importance in parallel operations
- and does not apply only to map!

#### The `map` Method of `Stream`
The map method of a `Stream<A>` takes a single argument
- the type is `Function<? super A, ? extends R>`
	- it accepts the stream's data as an argument
	- the return type is assignable to the downstream type
The resulting `Stream<R>` will contain
- the *same number* of items as the input stream
- which will be the values returned from the supplied function

#### The `filter` Method of `Stream`
The filter method of a `Stream<A>` takes a single argument
- the type is `Predicate<? super A>`
	- it accepts the stream's data as an argument
	- for each item in the stream, if the predicate returns `true`, that item is passed to downstream
The resulting `Stream<A>` will contain
- the *same or fewer* items as the input stream
- which will be unchanged values of the original stream

#### The `flatMap` Method of `Stream`
In a `map` operation, the number of output items is the same as the number of input items.

In a `filter` operation, the items are unchanged, but some might be omitted.

If a `map` operation returns an array, or a list, that is one item from the perspective of the stream

If we want one input item to result in an arbitrary number (zero, one, or many) of items in the stream, we must use a `flatMap` operation

The `flatMap` method on a `Stream<A>` takes a single argument
- the type is `Function<? super A, Stream<? extends R>>`
	- it accepts the stream's data as an argument
	- the returned value describes a stream of any number of items that are assignable to the downstream type 
	- each of those items becomes part of the resulting stream
	- boundaries related to the input item are absent from the resulting stream

In the `Stream<R>` returned by a call to `flatMap`
- the number of items is *unconstrained* by the number in the input stream
- the values are those of the streams returned from the supplied function

##### Quick Question
How are the operations `map`, `filter`, and `flatMap` related to these concepts?
- A loop
- Two nested loops
- A continue statement in a loop
Answer:
A `map` operation is comparable with a single loop.
A `filter` operation is comparable with a continue statement.
A `flatMap` operation is comparable with a pair of nested loop


If the input stream contains the strings "Fred", "Jim", "Sheila" and the desired output should contain F, r, e, d, J, i, m, S, h, e, i, l, a which operation must be present in the stream processing?

Answer: `flatMap`
This is the only method that can result in more items in the output than were in the input
