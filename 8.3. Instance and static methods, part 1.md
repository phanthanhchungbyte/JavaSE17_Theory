#### Method Declaration Syntax
Methods have a general declaration form:
`<modifiers> <header> <body>`
- modifiers are: annotations, `public, protected,private, abstract, static, final, synchronized, native, strictfp`
- header is: `[<generic-declarations>]<return type> <name> <formal parameter list> [<array>] [<throws>]`
- formal parameter list is zero or more formal parameters, comma separated, inside parentheses.

#### Method Header Syntax
The method header looks generally like this
```Java
static <E, F> void doStuff(String st, E e) throws SQLException
```
- The `[array]` part is deprecated syntax (borrowed from C):
```Java
int doOther()[]
```
which means exactly the same as:
```Java
int [] doOther()
```

#### Formal Parameter List Syntax
The method's formal parameter list looks generally like this:
```Java
void doStuff(@Deprecated String st, LocalDate date)
```
- The formal parameter list is in parentheses
- It has zero or more formal parameters, comma separated
- Positional argument passing only (not "named")
- No provision for default values
- Parameters may be marked `final`
- Parameter types may be annotated

#### The throws Clause
A method that could, or in future might need to, throw *checked exception* must declare this in a throws clause
```Java
void doStuff() throws AnException, AnotherException
```
The listed exceptions:
- may be one or more (omit throws entirely for zero)
- need not be possible in the method
- are presented comma separated
- may be repeated
- must include any unhandled checked exception that can arise in the code it, or a parent exception
- unchecked exceptions may be listed if desired

#### The Method Body
The method body is enclosed in curly braces
- Unless declared as a void method, the body must return an assignment-compatible value from all non-exception exit paths
	- i.e. all return statements must return the correct type
- Beware of "running off the end", which is only compatible with a void method

##### What happens?
```Java
final public String doStuff(final int x) {
	if (testIt(x)) {
		return x + " good";
	} else return x;
}
```
Answer: Compilation fails
`return x` is the wrong type
`final public` is OK< but not conventional order

#### Rules for Static Methods
- No `this` reference
- Static methods *can* access instance fields, but not using an `unqualified` form
- Binding is static / compile time
- Invoke with context defining the containing class
	- Unqualified in the same class
	- Qualified with the class name prefix
	- Qualified with an object reference (expect interface statics)

##### Quick Question
What ways can you invoke doStuff at line n1?
How would this change if method `fromHere` were static?
```Java
class Stuff {
	static void doStuff() {}
	void fromHere() {
		// line 1
	}
}
```
Answer:
- doStuff();
- Stuff.doStuff();
- this.doStuff();
- new Stuff().doStuff();

if `fromHere` were static, `this.doStuff()` would fail.

How do you fix this while keeping `doStuff` as a static method?
```Java
class Scratch {
	int x;
	static void doStuff() {
		System.out.println(x);
	}
}
```
Answer:
```Java
static void doStuff (Scratch sc) {
	System.out.println(sc.x);
}
```
or
```Java
System.out.println(new Scratch().x)
```
