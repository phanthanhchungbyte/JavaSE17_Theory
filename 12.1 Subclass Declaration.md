#### Subclass Declaration Syntax
`<regular class declaration> extends <parentclass>`
`[<implements clause>]`
- The `extends` keyword is followed by the parent class
- Only one immediate parent class is permitted
	- Multiple parents exist as a chain of ancestors
- if the `extends` clause is omitted the parent is implicitly `java.lang.Object`
- If an `implements` clause is used, the `extends` clause must precede it

#### Controlling Subclassing
Subclassing can be controlled
- A class that is marked `final` cannot be subclassed
- A class that has ***only private constructors*** can be subclassed  ***only*** by nested/inner classes
Methods in a subclass that have the same simple name as those in a parent class will be overloads if the argument type-sequence differs, or attempts to override if not
- Overloading methods are independent of one another

#### Rules for Overriding Instance Methods
If an instance method in a subclass has the same name and argument type-sequence as a non-private instance method in the parent, then it is an attempt to override. For this to succeed:
- The parent method must *not* be marked `final`
And for the overriding method:
- The accessibility must *not* be less
- A reference return type must be assignment compatible
- A primitive return type must be identical
- Checked exceptions declared by the overriding method must be expected from the parent method
- `@Override` may be used to verify overloading

#### Hiding Static Methods
If a static method in a subclass has the same name and argument type-sequence as a non-private static method in the parent, then it is similar to overriding. For this to succeed:
- The parent method must *not* be marked `final`
And for the subclass method:
- The accessibility must *not* be less
- A reference return type must be assignment compatible
- A primitive return type must be identical
- Checked exceptions declared by the overriding method must be expected from the parent method
- `@Override` may be used to verify overloading

#### Reminder of Additional Considerations
Recall that argument type sequence does not differentiate every aspect of the source code
- Generics are not distinguishable
	- `List<String>` and `List<LocalDate>` are equivalent due to type erasure
	- However, they will be rejected in attempts to override
- Arrays and variable length arguments are equivalent and will form valid overrides

Where overriding instance methods exist, invocation is *dynamic*; the behavior will be determined by the type of object on which a method is invoked. By contrast, where static method exist, the behavior is determined entirely at compilation time based on the prefix reference type.

##### Quick Question
<span style="color: #F0F8FF;">What happens?</span>
```Java
class P {
	P doStuff(int x, String s) {return null;}
}

class Q extends P {
	@Override
	Q doStuff(Integer x, String s) {return null;}
}
```
Answer: The annotation raises a compilation error, since the type sequence of the arguments is not identical

<span style="color: #F0F8FF;">What happens?</span>
```Java
public final class P {
	private int x = 99;
	P doStuff(int x, String s) {return null;}
}

class Q extends P {
	@Override
	Q doStuff(int x, String s) {return null;}
}
```
Answer: The compilation fails because class P is `final`, and cannot appear in an `extends` clause


<span style="color: #F0F8FF;">What rules apply to the `doStuff` method in class Q?</span>
```Java
class P {
	private int x = 99;
	P doStuff(int x) {return null;}
}

class Q extends P {
	@Override
	Q doStuff(int x, String s) {return null;}
}
```
Answer: The method is an overloading method, and therefore no rules constrain it relative to the `doStuff` method in the class P
