Pattern variables are initialized only when `instanceof` yields `true`
- The scope of these variables is accurately tracked to where they will have *definitely* been initialized
- Take care with complex boolean expressions

```java
public class ScopeExamples {  
  public static void main(String[] args) {  
    Object object = "";  
    if (object instanceof String s) {  
      System.out.println("In scope: " + s);  
    } else {  
      System.out.println("Not in scope");  
    }  
  
    if (object instanceof String s && s.length() > 5) {  
      System.out.println("In scope: " + s);  
    } else {  
      System.out.println("Not in scope");  
    }  
  
    // Can't use || here!  
//    if (object instanceof String s || s.length() > 5) {

//    } else {  
//      System.out.println("Not in scope");  
//    }  
  
    if (!(object instanceof String s) || s.length() > 3) {  
  
    } else {  
      System.out.println("s is " + s);  
    }  
  }  
}
```

#### Scope of Pattern Variables
The following logical operators are often used with patterns and can control scope of pattern variables:
- Conditional-and: `&&`
- Conditional-or: `||`
- Logical complement: `!`
- Conditional operator: `? :`
Scope is limited by normal rules and may also be controlled by:
- `if/else`
- `while, do/while`, C-style `for`
- Exceptions

```Java

    if (!(object instanceof String str1)) throw new IllegalArgumentException();  
    System.out.println(str1);
```

#### Generics in `instanceof`
In Java 11 `instanceof` would not test against a generic type
- Unless the type used only wildcards
	- e.g., this was the only permitted form for generic types:
		`obj instanceof List<?>`

Rather more useful generic types can be included now
- but only to the extent that the generic aspects can be validated at *compilation time*
- This extension allows the pattern match variable to be declared with useful generic type information

Tests using `instanceof` are rejected at compile time if impossible
```Java
String st = "";
if (st instanceof Number) {} // That's simply impossible
```
Similarly, generic types can now be included only to the extent that the generic aspects can be validated at compilation time
- If the basic type verification shows true, the source must make it definite that the generic aspects will also be valid
```Java
Collection<String> strings = new ArrayList<String>();
if (strings instanceof List<String> ls) {}
```
##### Quick Question
What happens?
```Java
Object obj = "Vegetables";
if (object instanceof String s || s.length() > 5) {
	System.out.println("A message");
} else {
	System.out.println("Other message");
}
```
Answer: this fails to compile, because `s` is not in scope on the right-hand side of the `||` operator

What is the scope of `s`?
```Java
Object obj = Math.random() > 0.5 ? "Fruits" : 5;
System.out.println(obj instanceof String s ? "A" : "B[]()");
```
Answer: When the `obj` actually is `instanceof String`, from the point of the declaration to the left of the colon (the region in which it can be true)
