#### Method References
Four patterns of lambda expression can be substituted with a modified syntax called a method reference
- These cannot express anything that can't already be expressed
- They might or might not involve less typing
- They entirely hide the argument passing, and prohibit any manipulation of arguments
- They focus on the behavior of the expression by ignoring the arguments
	- and emphasize the class containing that behavior
- Four types exist

#### Method Reference Forms
**This form can be used when a static method is to be invoked**
If the lambda would look like this
```java
(a, b) -> MyClass.doStuff(a, b)
```
then we can write this instead
```java
MyClass::doStuff
```

Notice the arguments have vanished, we entirely lose control of them, if the order in the lambda doesn't match the requirements of the method, we cannot use a method reference.

This form can work with lambdas requiring any number of arguments, provided they're right for the method to be called


**This form can be used when a constructor is to be invoked**
If the lambda would look like this
```java
(a, b) -> new MyClass(a, b)
```
then we can write this instead
```java
MyClass::new
```

The same comments about arguments apply here. Any number can be supported, provided they exactly match what's required for the constructor's argument list


**This form can be used when an instance method is to be invoked on the first argument of the lambda**
If the lambda would look like this
```java
(MyClass a, Blah b) -> a.doStuff(b)
```
then we can write this instead
```java
MyClass::doStuff
```

In this form, there must be at least one argument to the lambda, as there must be a context object for the method invocation. 
Any remaining arguments must exactly match what's required for the method's argument list.

**This form can be used when an instance method is to be invoked on an arbitrary object expression**
If the invocation could look like this
```java
final MyClass t = some ()[0].expression;
(a, b) -> t.doStuff(a, b)
```
then we can write this instead
```java
some()[0].expression::doStuff
```

In this form, the object expression need not be a simple variable but the expression is evaluated *exactly once prior* to potentially multiple invocations of the method reference's behavior.

#### More About Method References
If the method reference could resolve to more than one expansion, it will fail to compile.

If the arguments do not match in type and number, the expression will fail to compile--you cannot manipulate the arguments in any way!

A method reference must conform to the argument and return-type requirements of the context (just like the lambda it replaces)
