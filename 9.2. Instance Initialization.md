#### Initializing Instance Fields
Instance fields are initialized during construction
- To zero / zero-like values
- Using an assignment from an expression in the declaration
- In any number of instance initializer blocks
- In a constructor or constructors chained using `this(...)`
Notes
- Initializations execute in order from top to bottom
- Must declare before making an unqualified read
- Only an initializer block can express flow control
- No checked exceptions can be unhandled

##### What happens?
```Java
class TryIt {
	public static void main(String[] args) {
		new TryIt().go();
	}
	
	public void go() {
		System.out.println(x + ", " + y + ", " + z);
	}
	int x;
	{System.out.println(x + ", " + this.y);}
	int y = 100;
	{System.out.println(x + ", " + this.y)};
	{x = y + 10;}
	int z = -1;
}
```
Answer: 0, 0
0, 100
110, 100, -1

#### Initialization Using Constructors
Instance initialization can also be performed using *constructors* 
- These should perhaps be called "initializers", since they do not perform construction
When `new Xxx(...)` is invoked:
1. memory is allocated and zeroed (as if by new)
2. a constructor matching `Xxx(...)` is called

The constructor:
1. (generally) passes control to the superclass
2. runs the instance initializations

#### Constructor Declaration Syntax
The declaration of a constructor take the form
`[<modifiers>][<generic-declarations>].`
`<name>(<formal-parameter-list>)[<throws>]<body>`
- `<modifiers>` may be: annotations, `public`, `private`, and `protected`
- `<name>` *must be*  the classname
- `<formal-parameter-list>`, `<throws>`, and `<body>` are the same as for instance methods

Constructors may use `return;` but they behave as void methods that mutate their implicit `this` argument.

#### Delegating to Overloaded Constructors
Java allows overloaded constructors
- Argument type sequence must differ as for methods
- Delegate from one to another by invoking `this(...);`
- If delegation occurs, responsibility for superclass initialization is also delegated
	- Superclass initialization precedes instance initialization

All constructors *start* with one of:
- explicit call to `this(...);`
- explicit call to `super(...);`
- implicit call to `super(...);`

#### Constraints on Delegation Arguments
When delegating with `this(...)` or `super(...)`
- argument expressions *must not use* the `this` reference, either implicitly or explicitly
	- Java assumes the current object is not yet initialized.
	- Argument locals, static members, and external references are fine

#### The Default Constructor
If the source code does not declare any constructor
- The compiler creates the "default constructor"

A default constructor has these properties:
- Accessibility matches the class
	- Except for enums, which are always private
- Zero arguments (except for inner classes)
- No `throws` clause
- Delegate to `super()` with zero arguments
Note this requires the parent class have an accessible zero argument constructor

#### Exceptions in Constructors
If a constructor throws a checked exception
- this indicates that the object initialization failed
- it makes no sense to think that a subclass object can be valid if it's "parent aspect" is invalid
- also, `this(...)` or `super(...)` must be the first thing in any constructor, and try/catch cannot surround these
Because of this any checked exception thrown by a parent class constructor must be thrown, and declared, by the child.
- Unlike overridden methods, overloaded constructors cannot throw fewer exceptions than their parent constructors
