#### `record` Fundamentals
A `record` creates a restricted class with a simple syntax
- A `record` groups related values into a single container
	- An "aggregate type" or "data carrier"
	- Intended to be unmodifiable
- The simple syntax provides
	- Initialization
	- Accessor, but not setter, methods
	- Content based `equals` and `hashcode` methods
		- `null` values are handled sensibly
	- A simple, but meaningful `toString` method

#### Basic Syntax
The basic syntax for a `record` is very simple:
```Java
record Customer (String name, int creditLimit) {}
```

#### `record` Rules
For a `record`
- The parentheses around the component list are required
	- Even with zero components
- Curly braces for the class body are required
- The class is concrete, and implicitly final
	- May optionally be labeled as such
	- Subclassing could break immutability
- An extends clause is prohibited
	- Parent type is implicitly `java.lang.Record`
- Implementing interfaces is allowed
- No explicit instance fields may be declared
	- Member records are implicitly static, to prevent a mutable `Enclosing.this` field
	- Member records may be redundantly labeled `static`, unless they are local
- Components
	- Are stored as private, final, members with the same name and type
	- May be generic
	- The last specified component may hagve variable arity, which is represented as an array
	- A component's name cannot match a method of `Object`
- Accessor methods have the same name as the component
	- Not "`getXxx`" 
	- They are public
	- They take zero arguments
	- They throw no exceptions
	- Their return type is the type of the component

##### Quick Question
Given:
```Java
record Customer(String name, String... purchasedItems) {}
```
Describe the accessor method for `purchasedItems`

Answer:
```Java
public String[] purchasedItems() {
	return this.purchasedItems;
}
```
`==================================`
Given:
```Java
record Named<E>(String name, E item) {}
record NamedEvent(String name, LocalDate item) extends Named<LocalDate> {}
```
and:
```Java
NamedEvent ne = new NamedEvent (
	"New Century Day", LocalDate.of(1, 1, 2101));
var x = ne.item();
```
What do you notice about `var x`?

Answer: x is a distraction
Record types are implicitly final, so `NamedEvent` is invalid

`================================`
What do you notice?
```Java
interface FirstNameable {String first();}
interface FirstNameable {String first();}
record NamePair(String first, String last) implements FirstNameable, LastNameable {}
```
Answer: This example is valid, records can implement interfaces. Component access methods are public, zero argument, return the component type, declare no exceptions, and have the name of the component, so they are correct matches for these interfaces.


`===============================`
What's the result?
```Java
record Named(StringBuilder name) {}
```

and\:
```Java
Named n1 = new Named(new StringBuilder ("Fred"));
Named n1 = new Named(new StringBuilder ("Fred"));
System.out.println(n1.equals(n2));
n2.name().append("dy");
System.out.println(n2);
```
Answer: 
- false
- `Named[name=Freddy]`
`StringBuilder` does not provide a meaningful `equals`
It's also a mutable type so, like any final field, the record cannot prevent such a component's object from mutating
